프로젝트마다 매번 워크플로우 YAML을 새로 작성하는 대신, "이 스택에 맞는 CI/CD 만들어줘" 한 줄로 빌드·테스트·배포 파이프라인을 표준 구성으로 생성하는 스킬입니다.

## 파이프라인이 갖춰야 할 최소 구성

1. **트리거 조건** — 어떤 브랜치·이벤트에서 실행할지 (예: `main` 푸시, PR 생성)
2. **빌드 단계** — 의존성 설치, 정적 사이트/애플리케이션 빌드
3. **검증 단계** — [자동 코드 리뷰](../../security-and-quality/auto-code-review/), [테스트 자동 생성](../../security-and-quality/test-generation/)으로 만든 테스트, 보안 스캔
4. **배포 단계** — 검증을 통과한 경우에만 실제 배포 실행
5. **권한 최소화** — 워크플로우에 필요한 최소 권한만 부여 (`permissions:` 명시)

{{< callout type="info" >}}
GitHub Actions에서 `permissions: contents: read, pages: write, id-token: write`처럼 필요한 권한만 명시적으로 선언하면, 워크플로우 파일이 탈취되더라도 피해 범위가 제한됩니다.
{{< /callout >}}

## 흔한 함정

- **버전 고정 누락**: 빌드 도구 버전을 고정하지 않으면 어느 날 갑자기 빌드가 깨질 수 있음
- **캐시 미사용**: 매 빌드마다 의존성을 새로 받으면 파이프라인이 느려지고, 결국 사람들이 로컬 빌드로 우회하게 됨
- **검증 단계 생략**: 배포 속도를 위해 테스트·보안 스캔 단계를 건너뛰면 [훅으로 반복 작업 자동화](../../orchestration/hooks-automation/)에서 경고한 것과 같은 리스크가 그대로 파이프라인에 재현됨
