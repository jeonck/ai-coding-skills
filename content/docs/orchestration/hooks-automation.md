사람이 매번 "보안 스캔 돌려줘"라고 기억해서 요청해야 한다면, 언젠가 그 요청을 잊는 순간 취약점이 그대로 배포됩니다. **훅(Hook)**은 이런 "항상 일어나야 하는" 작업을 특정 이벤트에 묶어 시스템이 자동으로 실행하게 합니다.

## 흔히 훅으로 자동화하는 지점

| 이벤트 | 자동 실행 작업 |
|---|---|
| 커밋 전 (pre-commit) | 린트, 포맷, [VibeSec-Skill 취약점 스캔](../../security-and-quality/vibesec-owasp/) |
| PR 생성 시 | [자동 코드 리뷰 스킬](../../security-and-quality/auto-code-review/) 실행 |
| 배포 완료 후 | 헬스체크, 팀 채널 알림 |
| 도구 호출 전/후 | 위험한 명령어 차단, 실행 로그 기록 |

{{< callout type="info" >}}
훅은 "이 스킬을 언제 실행할지"를 사람의 기억이 아니라 시스템 이벤트에 위임하는 것입니다. 반대로 사람의 판단이 필요한 작업(예: [사이트 클론 코딩](../../clone-and-scaffold/clone-site-design/) 결과 검토)은 훅이 아니라 스킬로 남겨둬야 합니다.
{{< /callout >}}

## 설계 원칙

1. **실패를 조용히 넘기지 않기** — 훅이 실패하면 반드시 눈에 띄게 알림
2. **되돌릴 수 없는 작업은 훅에 넣지 않기** — 자동 배포, 강제 푸시처럼 위험한 작업은 훅이 아니라 명시적 승인 단계를 거치게 함
3. **실행 시간을 짧게 유지하기** — 훅이 느리면 커밋/PR마다 대기 시간이 쌓여 결국 우회(`--no-verify`)하고 싶은 유혹이 커짐

{{< callout type="warning" >}}
훅을 우회하는 습관(`--no-verify` 등)이 한 번 생기면 훅 자체가 무의미해집니다. 훅이 자주 우회된다면 훅을 없앨 게 아니라, 왜 느리거나 자주 실패하는지 원인을 먼저 점검하세요.
{{< /callout >}}
