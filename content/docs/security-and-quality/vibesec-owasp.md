"VibeSec-Skill을 이용해서 OWASP 관점 보안 취약점 해소"라는 한 줄 명령은, 코드를 짜는 것("vibe coding")과 그 결과를 안전하게 만드는 것("sec")을 하나의 스킬로 묶은 사례입니다. 내부적으로는 아래 절차가 고정되어 있습니다.

## 표준 절차

1. **정적 분석(SAST) 실행** — Semgrep, CodeQL 등으로 코드 패턴 기반 취약점 탐지
2. **OWASP Top 10 매핑** — 발견된 이슈를 A01(접근 제어)~A10(SSRF) 등 카테고리로 분류
3. **우선순위화** — 심각도(Critical/High/Medium)와 실제 도달 가능성(reachability) 기준으로 정렬
4. **자동 수정 제안** — 안전한 패턴(파라미터 바인딩, 이스케이프, 검증 로직)으로 교체
5. **회귀 테스트 확인** — 수정 후 기존 테스트가 여전히 통과하는지 확인

{{< callout type="info" >}}
{{< term "OWASP Top 10" >}}은 웹 애플리케이션에서 가장 흔하고 위험한 취약점 10개 범주입니다. 스킬이 "OWASP 관점"이라고 말할 때는 보통 이 카테고리 체계를 기준으로 이슈를 분류하고 우선순위를 매긴다는 뜻입니다.
{{< /callout >}}

## 자주 걸리는 취약점 카테고리와 수정 패턴

| OWASP 카테고리 | 흔한 원인 | 자동 수정 패턴 |
|---|---|---|
| A03 Injection | 문자열 결합으로 쿼리 생성 | 파라미터 바인딩/ORM으로 교체 |
| A01 Broken Access Control | 권한 검증 누락 | 미들웨어/가드에서 역할 검증 강제 |
| A02 Cryptographic Failures | 평문 저장, 약한 해시 | 표준 라이브러리의 강력한 해시·암호화로 교체 |
| A05 Security Misconfiguration | 기본 설정 그대로 배포 | 프로덕션 전용 보안 설정 템플릿 적용 |

{{< callout type="warning" >}}
자동 수정 스킬이 "취약점 0건"이라고 보고해도, 이는 스캐너가 아는 패턴 범위 내의 이야기입니다. 비즈니스 로직 취약점(예: 가격 조작, 권한 상승 경로)은 사람의 위협 모델링이 여전히 필요합니다.
{{< /callout >}}
