---
title: "테스트 자동 생성"
weight: 3
---

새 기능을 추가하거나 버그를 고친 뒤 "이 변경분에 대한 테스트 만들어줘"라는 한 줄 명령으로, 변경된 코드 경로를 검증하는 테스트 케이스를 생성하는 스킬입니다.

## 표준 절차

1. **변경 범위 파악** — diff에서 새로 추가/수정된 함수와 분기 확인
2. **엣지 케이스 도출** — 빈 입력, 경계값, 동시 요청, 예외 상황 나열
3. **기존 테스트 스타일 준수** — 프로젝트에 이미 있는 테스트 프레임워크·네이밍 규칙을 그대로 따름
4. **커버리지 확인** — 생성된 테스트가 실제로 변경된 라인을 실행하는지 커버리지 리포트로 검증

{{< callout type="info" >}}
좋은 테스트 생성 스킬은 "통과하는 테스트"가 아니라 "변경 전 코드로 돌리면 실패하는 테스트"를 목표로 합니다. 버그 수정이라면 수정 전 코드에 이 테스트를 돌려 실제로 실패하는지 먼저 확인하는 것이 검증의 핵심입니다.
{{< /callout >}}

## 흔한 실수

- 이미 통과할 수밖에 없는, 즉 아무것도 검증하지 않는 테스트를 생성하는 경우 (예: mock이 실제 로직을 모두 대체해버린 테스트)
- 프로젝트의 통합 테스트 관행을 무시하고 매번 새로운 방식(다른 mocking 전략 등)을 도입하는 경우

{{< callout type="warning" >}}
데이터베이스나 외부 API를 모두 mock 처리한 테스트는 "통과"해도 실제 운영 환경의 문제를 놓칠 수 있습니다. 프로젝트에 이미 통합 테스트 관행이 있다면 그 방식을 우선 따르세요.
{{< /callout >}}
