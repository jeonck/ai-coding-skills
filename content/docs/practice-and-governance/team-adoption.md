한 사람이 만든 편리한 한 줄 명령을, 팀 전체가 신뢰하고 쓰는 표준으로 만드는 과정에는 개인 작업과는 다른 단계가 필요합니다.

## 도입 단계

1. **개인 실험 단계** — 반복되는 작업을 개인이 스킬로 먼저 만들어 여러 번 검증
2. **공유 저장소로 승격** — 검증된 스킬을 팀 공용 저장소로 옮기고, 리뷰를 거쳐 병합
3. **표준 트리거 문구 합의** — 같은 스킬을 팀원마다 다른 말로 불러 혼선이 생기지 않도록 트리거 키워드를 통일
4. **온보딩 문서화** — 신규 팀원이 "어떤 스킬이 있는지" 검색할 수 있는 인덱스 유지 ([스킬 축적·버전관리 전략](../skill-versioning/) 참고)

{{< callout type="info" >}}
도입 초기에는 스킬 수를 늘리는 것보다 **가장 자주 반복되는 3~5개 작업**을 골라 확실하게 표준화하는 편이 효과가 큽니다. 이 로드맵에서도 [클론·생성](../../clone-and-scaffold/)과 [보안·품질](../../security-and-quality/) 두 축을 가장 먼저 다루는 이유가 여기에 있습니다.
{{< /callout >}}

## 저항이 생기는 지점과 대응

| 저항 | 원인 | 대응 |
|---|---|---|
| "내 방식이 더 편하다" | 스킬이 실제 반복 작업과 맞지 않음 | 스킬을 실제 작업 패턴에 맞춰 개정 |
| "결과를 못 믿겠다" | 스킬 실행 결과에 대한 검증 절차 부재 | 결과를 사람이 diff로 확인하는 단계를 스킬에 명시 |
| "매번 같은 실수를 반복한다" | 함정이 스킬 문서에 기록되지 않음 | 실패 사례를 스킬 문서에 즉시 추가 |

## 다음 단계

스킬 사용이 늘어날수록 되돌리기 어려운 작업(배포, 저장소 생성, 강제 푸시)에 대한 통제가 중요해집니다. [자동화 스킬의 안전장치와 리스크 관리](../safety-guardrail/)에서 이어서 다룹니다.
