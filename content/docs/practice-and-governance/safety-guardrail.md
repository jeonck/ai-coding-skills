스킬·훅·멀티에이전트로 자동화 범위가 넓어질수록, "빠르게 되는 것"과 "안전하게 되는 것" 사이의 균형이 중요해집니다. 이 페이지는 이 로드맵 전체에서 반복적으로 강조한 원칙을 하나로 정리합니다.

## 되돌리기 어려운 작업은 항상 사람이 확인

- 저장소 생성·공개 범위 변경 ([한 줄 배포](../../deploy-and-ops/one-click-deploy/))
- 강제 푸시, 히스토리 재작성
- 프로덕션 환경 설정 변경
- 외부 시스템에 메시지·티켓을 생성하는 MCP 도구 호출 ([MCP 연동](../../orchestration/mcp-integration/))

{{< callout type="warning" >}}
"한 줄 명령으로 다 된다"는 편리함이, "확인 없이 다 된다"를 뜻해서는 안 됩니다. 이 로드맵의 모든 스킬은 결과를 사람이 검토할 지점을 최소 하나는 남겨두도록 설계되어 있습니다.
{{< /callout >}}

## 자동 수정을 맹신하지 않기

[VibeSec-Skill로 OWASP 취약점 해소](../../security-and-quality/vibesec-owasp/)나 [자동 코드 리뷰](../../security-and-quality/auto-code-review/)처럼 자동으로 코드를 고치는 스킬은, 알려진 패턴 범위 안에서만 정확합니다. 비즈니스 로직 취약점, 팀 고유의 컨벤션 위반은 여전히 사람의 검토가 필요합니다.

## 감사 가능성(Auditability) 확보

1. **모든 자동 실행을 로그로 남기기** — 어떤 스킬/훅이 언제, 무엇을 변경했는지 추적 가능해야 함
2. **훅 우회 이력 확인** — `--no-verify` 같은 우회가 반복된다면 훅 자체의 속도·정확성 문제를 먼저 점검
3. **정기적인 스킬 재검토** — [스킬 축적·버전관리 전략](../skill-versioning/)에서 다룬 것처럼, 오래된 스킬이 현재 정책과 어긋나지 않는지 주기적으로 재확인

{{< callout type="info" >}}
여기까지가 여섯 개 카테고리의 핵심 원칙입니다. 직접 스킬을 만들어보고 싶다면 [실습 (Labs)](../../../labs/)에서 클론 코딩·보안 스캔·배포 자동화 스킬을 순서대로 따라 해보세요.
{{< /callout >}}
