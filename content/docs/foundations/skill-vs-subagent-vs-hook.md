title: "스킬 vs 서브에이전트 vs 훅"

---

같은 "한 줄로 자동화한다"는 목표라도, 실제로 이를 구현하는 방식은 세 가지로 나뉩니다. 이 셋을 구분하지 못하면, 병렬 실행이 필요한 작업을 스킬 하나에 욱여넣거나, 매번 사람이 호출해야 할 작업을 훅으로 만들어 오히려 통제력을 잃는 실수를 하게 됩니다.

## 비교표

| | 스킬 (Skill) | 서브에이전트 (Subagent) | 훅 (Hook) |
|---|---|---|---|
| 실행 위치 | 현재 대화 컨텍스트 안 | 별도의 독립된 컨텍스트 | 이벤트 발생 시 자동 |
| 호출 방식 | 사람이 이름을 불러 트리거 | 메인 에이전트가 위임 | 트리거 조건 충족 시 시스템이 실행 |
| 대표 용도 | 반복되는 단일 워크플로우 (클론 코딩, 보안 스캔) | 컨텍스트를 많이 소모하는 조사·리서치, 병렬 작업 | 커밋 전 린트, 배포 후 알림처럼 "항상 일어나야 하는" 작업 |
| 결과 가시성 | 대화 안에 그대로 노출 | 요약만 메인 대화로 전달 | 로그/알림으로만 확인 |

## 언제 무엇을 선택해야 하는가

- **스킬**을 선택하는 경우: "사이트 클론 코딩"처럼 사람이 판단하며 지켜봐야 하는 작업. 결과를 바로 검토하고 다음 지시를 이어가야 할 때.
- **서브에이전트**를 선택하는 경우: 조사에 필요한 중간 산출물(로그, 검색 결과)이 많아서 메인 대화 맥락을 오염시키고 싶지 않을 때. [멀티에이전트 병렬 실행](../../orchestration/multi-agent-parallel/)에서 자세히 다룹니다.
- **훅**을 선택하는 경우: 사람이 매번 기억해서 요청하기보다, 특정 이벤트(커밋, 배포)마다 무조건 실행되어야 하는 검증·통보 작업. [훅으로 배포 자동화](../../orchestration/hooks-automation/)에서 다룹니다.

{{< callout type="warning" >}}
셋을 혼동해 "훅"으로 만들어야 할 보안 검사를 사람이 매번 스킬로 호출하게 두면, 언젠가 호출을 잊는 순간 취약점이 그대로 배포됩니다. 반대로 사람의 판단이 필요한 클론 코딩 작업을 훅으로 자동화하면 결과를 검토할 타이밍을 놓칩니다.
{{< /callout >}}
