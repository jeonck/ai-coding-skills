title: "한 줄 명령이 동작하는 원리"

"사이트 디자인 한 번에 클론 코딩"처럼 짧은 한 줄 명령이 실제로는 수십 단계의 작업(페이지 구조 분석 → 컴포넌트 추출 → 스타일 토큰화 → 코드 생성 → 빌드 확인)을 안정적으로 수행합니다. 이것이 가능한 이유는 명령어 자체가 짧을 뿐, 뒤에서는 세 가지 메커니즘이 함께 작동하기 때문입니다.

## 1. 트리거링 (Triggering)

에이전트는 사용자 입력의 키워드·의도를 미리 등록된 스킬 목록과 매칭해, 가장 적합한 스킬을 자동으로 선택합니다. "이 사이트처럼 만들어줘"라는 말이 들어오면 클론 코딩 스킬이, "보안 취약점 점검해줘"라는 말이 들어오면 보안 스킬이 트리거됩니다.

## 2. 컨텍스트 주입 (Context Injection)

스킬이 트리거되면, 대화창에는 보이지 않는 **절차 문서 전체**가 에이전트의 작업 맥락에 주입됩니다. 사용자는 한 줄만 입력했지만, 에이전트는 그 순간 수십~수백 줄의 체크리스트와 예외 처리 규칙을 참고하게 됩니다.

{{< callout type="info" >}}
바로 이 지점이 [스킬 vs 서브에이전트 vs 훅](../skill-vs-subagent-vs-hook/)에서 설명하는 세 메커니즘의 공통 기반입니다 — 셋 다 "미리 준비된 맥락을 필요한 순간 주입한다"는 점은 같고, 실행 단위(같은 대화 안 vs 별도 에이전트 vs 이벤트 트리거)만 다릅니다.
{{< /callout >}}

## 3. 결정적 워크플로우 (Deterministic Workflow)

좋은 스킬은 "알아서 잘 해줘" 대신 **순서가 고정된 단계**로 작성됩니다. 예를 들어 클론 코딩 스킬은 "1) 대상 URL 접속 → 2) DOM/스타일 추출 → 3) 컴포넌트 단위 분해 → 4) 코드 생성 → 5) 로컬 빌드로 검증"처럼 순서가 명시되어 있어, 실행할 때마다 결과 품질의 편차가 줄어듭니다.

## 왜 이게 "생산성"과 "안정성"을 동시에 높이는가

- **생산성**: 매번 프롬프트를 새로 설계할 필요 없이 한 줄 호출로 끝나므로 반복 작업의 소요 시간이 줄어듭니다.
- **안정성**: 절차와 주의사항이 문서로 고정되어 있으므로, 실행자의 숙련도와 무관하게 결과 품질이 일정 수준 이상 유지됩니다.
